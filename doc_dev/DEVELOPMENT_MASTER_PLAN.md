## 一、 产品功能规格 (Product Feature Specification)
接手团队必须知道，用户最终看到的产品长这样：

1. **沉浸式古籍阅读器 (The Reader Layer)**
   - 功能：原生纯净阅读，支持原典/影印双排对照，自动排版长对话缩进。
   - 高级：20+ 类实体（人地事物）多色语法高亮，段落具有唯一锚点 (Anchor) 可精准分享。
2. **无限时空 3D 原地沙盘 (The Spatial Layer)**
   - 功能：基于 3D DEM 高程地形。
   - 交互：支持时间滑块过滤数据，提供全景层、人物足迹层（动态流光）、家族迁徙网、战争热力柱、以及自然生态变迁（多边形显隐）。
3. **线性时间图谱 (The Timeline Layer)**
   - 功能：类"地铁图"形态的历史进程纵览。
4. **底层图谱支撑 (The Graph Backend)**
   - 功能：实体消歧聚合、历史节点搜寻。

## 二、 系统总体架构与数据流流向 (Architecture)

坚持"离线自动化工厂"与"纯静态前端"物理隔离。

### 2.1 统一九步处理管线 (Data Pipeline)
依靠 Python 调度 Claude Agent，完成数据精炼：
```text
[PDF/TXT输入] 
   │
   ▼
1.预处理与校勘 (清理杂字符)
   │
   ▼
2.结构分析 (打上段落唯一编号定位锚点)
   │
   ▼
3.实体构建 (给原文打上标注，如〖=太原〗)
   │
   ▼
[4.事件提取 / 5.实体关系构建 / 6.本体挂载]  <-- 并行操作
   │
   ▼
7.逻辑推理与时空挂载 (将"太原"匹配到历史 GIS 库的 [112.5, 37.8] 经纬度，排查时间穿帮)
   │
   ▼
8.质量拦截 (Lint校验原文字符完整性)
   │
   ▼
9.前端 JSON 序列化输出
```

### 2.2 前端在线引擎
`JSON数据表 -> 浏览器 Vue3 内存 -> MapLibre GL渲染底图 -> Deck.gl 绘制上层发光特效`
**重要决策**：为保证 100% 零门槛开源，彻底抛弃需要 API Key 的 Mapbox，全系转向纯开源免费的 **MapLibre GL JS**。

### 2.3 古地名沿革消歧机制 (Toponym Evolution & Resolution)
不同朝代对同一片区域的称呼往往完全不同（如"建康"→"江宁"→"应天府"→"南京"），**严禁让大模型去脑补坐标映射**。本项目的消歧架构如下：
1. **AI 只做文字剥离**：大模型在管线第 3 步中，仅负责准确标出 `〖=建康〗`（并记录其上下文中的朝代信息），不做任何空间推断。
2. **CHGIS 词典做底层缝合**：在管线第 7 步中，Python 脚本携带 `[地名: 建康, 朝代: 晋]` 去查询 CHGIS 历史地理数据库，获取唯一的 `geo_id` 与经纬度坐标。由于"建康（晋）"和"江宁（清）"在 CHGIS 中指向同一个 `geo_id`，它们在最终 JSON 里会被绑定到完全相同的物理坐标。
3. **前端时空变幻展示**：当用户在浏览器中拖动时间滑块，同一块区域上悬浮的地名标签会随时间流转而平滑切换（"建康" → "江宁"），侧边栏可展示该地点的完整多朝代身世链。

## 三、 核心数据契约设计 (Data Schema Contracts)

前后端协作必须死守此契约，不得擅改字段名。

### 3.1 实体规范库 (`persons.json`, `families.json`)
```json
// persons.json 示例
{
  "pid_1001": {
    "name": "苏轼", "dynasty": "Song",
    "track_points": [ // 时空足迹数组，供 Deck.gl TripsLayer 使用
      {"year": 1037, "lng_lat": [103.83, 30.04], "location": "眉州", "event_ref": "evt_991"}
    ]
  }
}
```

### 3.2 地理演变库 (`geo_changes.json` 或 `PMTiles`)
负责处理沧海桑田。
```json
{
  "gid_502": {
    "name": "云梦泽",
    "polygon": [[112.1, 29.5], [112.5, 29.8], ...],
    "appear_year": -500, // 形成或有记录期
    "fade_year": 1200    // 消退消失期，供前端依据时间条显隐
  }
}
```

## 四、 方志专属本体实体标注规范 (Gazetteer Ontological Annotation Spec)

针对地方志的学术特殊性（如卷轶浩繁的《建置志》《食货志》《人物志》《灾异志》等），我们摒弃传统 NLP 粗糙的 "人/地/组" 三分法，制定以下**高精度、多维度的方志专属本体标注体系**。所有的 AI Agent 必须使用这套核心符号系统执行细化任务：

### 4.1 🧔 人物社群类 (Persons & Social)
*来源篇目：《人物志》《列女传》《户口》等*
- **重要人物**：`〖@群体:名字〗`（必须带有具体细分，如 `〖@乡贤:王守仁〗`, `〖@名宦:苏轼〗`, `〖@列女:王氏〗`, `〖@仙释:法海〗`）
- **宗族/氏族**：`〖&氏族〗`（用于做家族迁徙网络，如 `〖&太原王氏〗`, `〖&琅琊诸葛氏〗`）
- **人口/户籍**：`〖%户口〗`（用于经济民生统计，如 `〖%丁男三万〗`, `〖%匠户〗`）

### 4.2 🗺️ 地理建置类 (Geography & Infrastructure)
*来源篇目：《建置志》《舆图》《疆域》等*
- **行政区划**：`〖=区划〗`（随朝代变迁的建制，如 `〖=江夏郡〗`, `〖=宛平县〗`）
- **山川/水系**：`〖~自然〗`（湖泊、河流、名山，如 `〖~云梦泽〗`, `〖~太行山〗`）
- **城池/关津**：`〖♜防区〗`（拥有防御属性的地理实体，如 `〖♜雁门关〗`, `〖♜风陵渡〗`, `〖♜洛阳城墙〗`）
- **街巷/村镇**：`〖⌂聚落〗`（微观居住地，如 `〖⌂乌衣巷〗`, `〖⌂杏花村〗`）

### 4.3 🏛️ 机构建筑与礼教类 (Institutions & Architecture)
*来源篇目：《官师志》《学校志》《秩祀志》等*
- **治所/公署**：`〖#衙署〗`（行政办公建筑，如 `〖#兰考县衙〗`, `〖#两广总督府〗`）
- **学校/书院**：`〖^教育〗`（科举教育实体，如 `〖^岳麓书院〗`, `〖^国子监〗`）
- **祠庙/陵墓**：`〖†宗教神祀〗`（祭祀与风水建筑，如 `〖†关帝庙〗`, `〖†炎帝陵〗`）
- **官职/功名**：`〖§职位〗`（人物的附加属性，如 `〖§知府〗`, `〖§三甲巡抚〗`, `〖§进士〗`）

### 4.4 🌾 食货经济类 (Economy & Resources)
*来源篇目：《食货志》《物产志》《盐法》等*
- **物产/特产**：`〖$物产〗`（用作物产分布地图，如 `〖$蜀布〗`, `〖$井盐〗`, `〖$建州茶〗`）
- **土地/赋役**：`〖¥课税〗`（如 `〖¥一条鞭法〗`, `〖¥秋粮〗`, `〖¥丁银〗`）

### 4.5 📜 纪事与灾祥类 (Events & Phenomena)
*来源篇目：《艺文志》《兵事志》《灾祥志》等*
- **兵变/战役**：`〖⚔兵事〗`（武装冲突，如 `〖⚔赤壁之战〗`, `〖⚔黄巾起义〗`）
- **灾象/祥瑞**：`〖!异象〗`（极其重要的环境变迁数据，如 `〖!大疫〗`, `〖!飞蝗蔽日〗`, `〖!景星见〗`）
- **经籍/碑刻**：`〖《文献》〗`（记载于方志中的文章书籍，如 `〖《滕王阁序》〗`, `〖《三字经》〗`）

### 4.6 ⏳ 时间锚点 (Chronology)
- **纪年/干支**：`〖⌚时间〗`（绝对时空锚准器，如 `〖⌚万历十五年〗`, `〖⌚庚申岁〗`）

### 5.2 Git 隔离与外部分发策略
`TotalData/` **整体被 `.gitignore` 排除在 Git 版本控制之外**。Git 仓库只追踪纯代码（`app/`、`scripts/`、`skills/`）和轻量级文档。

数据的分发遵循以下机制：
1. **`.gitignore` 物理隔墙**：在项目根目录的 `.gitignore` 中写入 `TotalData/`，确保无论本地 `TotalData/` 中有多少 GB 的原始书籍和 GIS 数据包，都绝对不会被推送到云端仓库。
2. **外部存储分发**：将 `TotalData/` 的完整内容压缩打包后，上传至以下任意一种或多种外部平台：
   - **Hugging Face Datasets**：适合大规模学术数据集的托管与版本管理。
   - **GitHub Release 附件**：将压缩包作为版本发布页的附件资源挂载。
   - **网盘/OSS 直链**：百度网盘、阿里云 OSS 等备选方案。
3. **README 下载指引**：在 Git 仓库的 `README.md` 和 `TotalData/README.md` 中，附上明确的下载链接和放置路径说明。用户 `git clone` 后，只需按指引下载数据包，解压到 `TotalData/` 路径下，即可完整运行整个项目。

### 5.3 数据生命周期流转图
```text
用户手动下载 TotalData 压缩包
        │
        ▼
解压到 project-root/TotalData/
        │
        ├── OriginalData/   ← 用户也可以自行投喂新书籍
        ├── GISData/        ← CHGIS 等外部库（用户按指引下载放入）

        │

        │
        ▼  [前端 Vue3 直接读取 ProcessedData/ 下的 JSON]
        │
    浏览器 3D 沙盘渲染
```

## 六、 开发规划、阶段与排期 (Phases & Estimations)

## 七、 部署、测试与准入质量标准 (QA & Release)

### 7.1 开发环境一键搭建 (Local Setup)
新成员 Onboarding 手册：
1. **环境准备**：要求 `Node.js 18+`, `Python 3.10+`。
2. **拉取代码**：`git clone` 仓库（此时 `TotalData/` 目录为空）。
3. **下载数据**：按 `README.md` 中的链接下载 `TotalData` 压缩包，解压到项目根目录。
4. **拉取依赖**：在 `app/` 目录下执行 `npm install`。
5. **启动应用**：执行 `npm run dev`，访问 `localhost:5173`。无 API Key 负担，立刻可用。
6. **数据引擎**：在项目根目录 `pip install -r requirements.txt`。运行 `python scripts/test_pipeline.py` 测试 AI 链路。

### 7.2 测试策略矩阵 (Quality Matrix)
- **数据管线测试**：不验证语义正确性（交给人工审核），但 **100% 验证文本未受损率** 和 **JSON Schema 字段完整性**。失败则阻止 `git commit`。
- **前端渲染测试**：Chrome/Edge 的高并发抗压测试。要求 10,000 个时空节点 (TripsLayer) 在中配笔记本上保持 60 FPS 刷新率。

### 7.3 分支与开源发布流 (Git Process)
- **分支原则**：`main` 始终保持可随时一键发布的纯净度；`feature/skills` 修改大模型指令；`feature/web` 修改渲染代码。
- **发布机制**：使用 Vercel 或 GitHub Pages。由于本项目为"零后端架构"，任何推送到 `main` 的代码，触发静态 Web 页打包构建即可完成全球上线部署。
- **数据发布**：每当 `TotalData/ProcessedData/` 有重大更新时，重新打包上传至 Hugging Face / GitHub Release，并更新 `README.md` 中的下载链接与版本号。

## 八、 防崩坏最高接手协议 (The Non-Negotiables)
1. **绝对禁止 ORM 和复杂数据库库引入**。Vuex/Pinia 只能存用户皮肤设置和登录信息，**1GB 的地缘坐标数组属于纯内存级数据，禁止放进响应式树中**。
2. **严防代码篡改史料**。发现大模型发挥过度修改了古文的一个标点，不许改代码去原谅它，必须修改 Skill 文档教育 AI，从源头切断。
3. **数据目录不可变更**。所有数据必须严格存放在 `TotalData/` 统一目录下，禁止在项目根目录或 `app/` 中散落数据文件。
